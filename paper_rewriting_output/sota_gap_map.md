# SOTA Gap Map

## Candidate Contribution vs. SOTA Landscape

| Candidate Contribution | What SOTA Already Does | User Evidence | Real Gap | Claim Strength | Risk |
|---|---|---|---|---|---|
| Reward generation → reward search | EUREKA: LLM generates reward code, in-context evolutionary optimization. Text2Reward: generates dense rewards from language. CARD: dynamic feedback for refinement. | EG-RSA maintains a structured reward schema and performs sequential, auditable edits across 10 iterations with memory. | Existing methods treat reward design as generation/refinement of code; none model it as a schema-based search process with memory, attribution, and rollback. | Strong — the structural contrast is clear and mechanism-level. | Medium — EUREKA's "iterative optimization" can be framed as search; the distinction rests on schema representation + memory, not just the word "search." |
| Semantic role attribution | All SOTA methods treat reward components as opaque terms; EUREKA uses raw reward values for feedback. | EG-RSA labels each component (dense_guidance, stability_quality, terminal_success, safety_constraint, control_cost) and computes dominant-component ratios from trajectory data. | No prior LLM reward design work attributes which reward role drives policy behavior; attribution exists in RL interpretability but not coupled to reward editing. | Medium — attribution is a novel mechanism in this context but not a standalone contribution without the editing loop. | Low — attribution is diagnostic; it does not overclaim. |
| Outcome lesson memory | Reflexion: stores verbal reflections. Voyager: stores skill code in a library. Both use memory for agent behavior, not reward design. | EG-RSA stores structured lessons (schema_diff, metric_delta, failure_modes, rollback_decision) as effective_edit_lesson and regression_lesson, retrievable by similarity. | No prior work stores both successful and failed reward edits as structured, retrievable lessons for future reward search. | Strong — this is a clear mechanism contribution with implementation evidence. | Low — the mechanism is implemented and used across iterations. |
| Operator-constrained editing | All LLM reward methods allow the LLM to emit free-form reward code (EUREKA, Text2Reward, CARD). | EG-RSA restricts LLM output to a JSON edit_plan (increase_weight, decrease_weight, add_component, add_event_rule, disable_component) compiled by trusted logic. | Constraining LLM reward design to auditable operators is unique; prior work trusts LLM code output more. | Medium-strong — the constraint is novel but the operators are relatively simple. | Low — the mechanism is demonstrated. |
| Risk-aware audit and rollback | Reward hacking literature (Amodei et al. 2016, Pan et al. 2022) identifies the problem but does not propose LLM-integrated audit mechanisms. | EG-RSA runs behavior risk audit (medium/high risk classification), scale audit, and automatic rollback on audit failure. Experiment shows audit deadlock → relaxed policy. | No prior LLM reward design work includes an explicit audit-repair-rollback loop. | Strong — this directly addresses a known safety gap. | Medium — the audit is rule-based, not learned; reviewers may ask about scalability of rules to new environments. |

## Gap Summary

**Gap 1 (strongest): Structured reward search with memory.** Current SOTA treats reward design as generation + iterative refinement. No method maintains a versioned reward schema, constrains edits to auditable operators, and stores both effective and regressive outcomes as structured memory. This is a mechanism-level gap, not a performance gap.

**Gap 2: Integrated audit-and-rollback for LLM reward design.** While reward hacking is well-studied, no prior LLM reward design method integrates explicit risk auditing and automatic rollback into the search loop. The audit-deadlock → relaxed-policy finding further shows that this gap has non-trivial design implications (the safety-exploration tradeoff in reward search).

**Recommendation:** Build the paper around Gap 1 as the controlling contribution, with Gap 2 as the strongest supporting mechanism. Attribution and memory are sub-components of the search framework, not standalone contributions.

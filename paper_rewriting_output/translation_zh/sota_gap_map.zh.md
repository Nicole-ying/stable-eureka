# SOTA 空白图（中文）

## 候选贡献与 SOTA 对比

| 候选贡献 | SOTA 已有成果 | 用户证据 | 真实空白 | 论断强度 | 风险 |
|---|---|---|---|---|---|
| 经验引导的奖励 schema 搜索 | EUREKA、Text2Reward、CARD 进行迭代奖励代码生成 | E1 实验数据展示跨 8+ 迭代的 schema 演化 | 当前方法没有跨迭代维护结构化奖励表示；EG-RSA 持久化版本化 schema 并从中搜索 | 中等 | 如果 schema 搜索不优于代码生成，范式论断将崩溃 |
| 语义角色归因 | EUREKA 使用标量反馈；无奖励代码组件诊断 | E3 角色标注数据展示 `r_approach_region` (0.42→主导) 和 `r_landing_quality` 占比 | LLM 奖励设计中不存在逐组件奖励归因 | 高 | 因果论断需要消融；标量反馈可能已足够 |
| 结构化结果教训记忆 | Reflexion、Voyager 使用语言/技能记忆，但不用于奖励编辑 | E4 教训卡片展示有效编辑和回归编辑均被存储 | 结构化奖励编辑记忆（差异 + 增量 + 失效模式）是新的 | 中等 | 记忆有益的证据是相关性的，而非因果性的 |
| 算子约束奖励编辑 | 所有现有方法使用自由形式代码生成 | E6 实现展示 5 个算子活跃 | 自由代码生成 → 5 个可审计编辑算子是新的 | 高 | 算子约束可能限制 LLM 创造力 |
| 带死锁分析的集成风险审计 | 事后安全论文存在（Amodei、Pan、Skalse），但无集成奖励搜索审计 | E5 审计展示严格审计死锁（Iter 2-8 停滞）；E2 展示放宽审计突破 | 内部奖励搜索循环中不存在风险感知审计 | 高 | 死锁可能特定于 LunarLander + 审计规则 |

## 空白总结

**主要空白 1：结构化奖励编辑记忆与语义角色归因。** 现有 LLM 奖励设计方法以自由形式代码在每个迭代生成奖励，仅以标量分数评估输出。没有一个维护带有语义标签和结构化结果教训的可审计 schema。这限制了迭代效率（每次从零生成）和可解释性（无法诊断哪个奖励组件导致失败）。

**主要空白 2：带经验证据的集成审计。** 安全论文存在但不与奖励设计循环集成。EG-RSA 的审计死锁发现——严格安全门控可能造成抑制改进探索的障碍——是集成时才出现的新兴属性，在当前事后安全文献中不存在。该发现指出了奖励搜索中安全与探索之间的根本性设计张力。

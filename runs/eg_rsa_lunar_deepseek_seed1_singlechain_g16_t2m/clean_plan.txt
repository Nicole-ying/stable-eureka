## Search Plan for Lunar Lander Reward Function

### Phase 1: Component Design and Tuning
1. **Distance Reward**: Start with `-distance * k1` where distance = sqrt(next_obs[0]^2 + next_obs[1]^2). Tune k1 to balance with other components (suggested range: 1-5).
2. **Velocity Penalty**: Start with `-speed * k2` where speed = sqrt(next_obs[2]^2 + next_obs[3]^2). Tune k2 to be slightly larger than k1 (suggested range: 2-10).
3. **Angle Penalty**: Start with `-abs(next_obs[4]) * k3`. Tune k3 to be moderate (suggested range: 1-5).
4. **Fuel Efficiency**: Start with `-0.1` when action == 2. Tune if needed (suggested range: 0.05-0.3).
5. **Landing Bonus**: Start with `+100` when done and landing conditions met. Tune up to +500 if agent struggles to land.
6. **Crash Penalty**: Start with `-100` when done but landing conditions not met. Tune down to -500 if agent crashes too often.

### Phase 2: Scaling and Balance
- Ensure all components have comparable expected magnitudes (within factor of 10).
- The distance reward should dominate early exploration, while landing bonus should be the primary terminal signal.
- Velocity and angle penalties should be significant enough to shape behavior but not dominate.

### Phase 3: Testing Strategy
1. Test with random agent to ensure baseline behavior is reasonable (not too positive or negative).
2. Test with heuristic policy (main engine when above pad, orientation engines to stay upright).
3. Iterate on component weights based on observed behavior.

### Phase 4: Edge Cases and Robustness
- Ensure crash_penalty and landing_bonus are mutually exclusive (use done flag to determine which applies).
- Verify that the reward doesn't encourage hovering indefinitely (fuel penalty should discourage this).
- Check that angle penalty doesn't prevent necessary tilting for horizontal movement.

### Phase 5: Final Validation
- Run multiple seeds to ensure consistent performance.
- Verify that the reward function produces learning curves that converge to successful landing policies.

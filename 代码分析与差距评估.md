# 代码分析与差距评估：FDRE-HRDC vs LLM Agent 奖励函数自进化

## 研究目标

构建一个 **LLM Agent 自主完成强化学习奖励函数的发现、验证、迭代、优化**系统，全过程不需要人类注入领域知识，不靠硬编码。

LLM 自己观察环境 → 自己理解任务 → 自己提出奖励函数 → 自己跑实验 → 自己分析结果 → 自己改进 → 最终产出可迁移的奖励函数。

---

## 当前代码与目标的差距

以下逐文件指出硬编码位置、人工介入点，以及 Agent 模式应有的做法。

---

### 1. 环境语义 — 硬编码于 `trainer.py:193-243`

**文件**: `code/src/llm_reward_evolver/trainer.py`
**函数**: `known_env_description()`

```python
if name.startswith("lunarlander"):
    return "Observation semantics for LunarLander-v3: obs = [x_position, y_position, ...]"
if name.startswith("acrobot"):
    return "Observation semantics for Acrobot-v1: obs = [cos(theta1), sin(theta1), ...]"
if name.startswith("mountaincar"):
    return "Observation semantics for MountainCar-v0: obs = [position, velocity]..."
```

- **当前实现**: 作者查文档手工编写了每个环境的观测维度语义说明，通过 if-else 分支匹配环境名。
- **应有实现**: Agent 读取 Gymnasium 文档、环境源码，或通过与环境的交互（reset/step）来自主推断每个观测维度的物理含义和取值范围。环境语义是 Agent 自己探索出来的，不是人类预埋的字符串。

### 2. 环境诊断知识 — 硬编码于 `evolver.py:127-179`

**文件**: `code/src/llm_reward_evolver/evolver.py`
**方法**: `RewardEvolver._build_feedback()`

```python
if self.config.env_name.lower().startswith("mountaincar"):
    feedback += "\nEnvironment-specific diagnostic for MountainCar:\n- Encourage progress toward the right goal position..."
if self.config.env_name.lower().startswith("acrobot"):
    feedback += "\nEnvironment-specific diagnostic for Acrobot:\n- Reward current high tip position and useful swing-up progress..."
if self.config.env_name.lower().startswith("lunarlander"):
    feedback += "\nEnvironment-specific diagnostic for LunarLander:\n- Reward slow, stable descent: penalize abs(x_velocity)..."
```

- **当前实现**: 三个环境各有一段作者手写的诊断建议，硬编码在代码里。每轮进化时拼接到 feedback 末尾发给 LLM。
- **应有实现**: Agent 拥有 `read_training_log()` / `analyze_trajectory()` 等工具，自主分析训练曲线、状态轨迹分布、失败时的状态特征，从数据中发现 "agent 高速撞地 → 应增加速度惩罚" 这类结论，并将发现写入新的奖励函数。诊断知识是 Agent 从数据中推理出来的，不是作者凭经验预写的。

### 3. Prompt 中的环境规则 — 硬编码于 `prompts.py:104-129`

**文件**: `code/src/llm_reward_evolver/prompts.py`
**函数**: `_environment_reward_rule()`

```python
if name.startswith("lunarlander"):
    return "- Bad quantities must reduce reward: distance, speed, tilt..."
if name.startswith("acrobot"):
    return "- Reward current high tip position and useful swing-up progress..."
```

- **当前实现**: 每个环境有专属的规则注入 prompt，告诉 LLM 注意什么不该做。
- **应有实现**: 这些规则应该是 Agent 经过多轮试错后**自己学到的**。第一轮失败后、第二轮优化时，Agent 回顾历史实验的失败模式，自己生成约束规则，而不是从 prompt 模板里继承人类经验。

### 4. HRDC 子目标定义与系数 — 硬编码于 `canonical_rewards.py:4-22`

**文件**: `code/src/llm_reward_evolver/canonical_rewards.py`
**变量**: `LUNARLANDER_FDRE_HRDC`

```python
progress = (abs(px)+abs(py)) - (abs(x)+abs(y))
centered = max(0.0, 1.0 - abs(x))
slow     = max(0.0, 1.0 - abs(vx) - abs(vy))
upright  = max(0.0, 1.0 - abs(angle) - 0.5*abs(av))
contact  = float(left) + float(right)
both     = 1.0 if left>0.5 and right>0.5 else 0.0
land     = 1.0 if original_reward>70.0 else 0.0
crash    = 1.0 if original_reward<-70.0 else 0.0
fuel     = 0.012 if int(action)==2 else (0.004 if int(action) in (1,3) else 0.0)
```

- **当前实现**: 作者基于 RL reward shaping 学术经验，手工分解了 8 个子目标（progress/centered/slow/upright/contact/both/terminal/fuel）。每个子目标的公式、阈值（`abs(angle) - 0.5*abs(av)`、`left>0.5`）、动作成本（`0.012 / 0.004`）全是人工设计。
- **应有实现**: Agent 通过观察环境任务、分析状态空间、与任务目标交互，**自主分解**出有意义的子目标，自主定义每个子目标的数学形式，自主设定超参数。`contact` 和 `both` 这种概念是 Agent 从数据中归纳出来的，不是人类预先告诉它的。

---

### 5. 早晚期动态权重划分与系数 — 硬编码于 `canonical_rewards.py:18-21`

```python
early = 0.04*progress + 0.04*centered + 0.03*upright
late  = 0.02*progress + 0.05*centered + 0.06*slow + 0.06*upright + 0.08*contact + 0.12*both
shaping = (1.0-p)*early + p*later
```

- **当前实现**: 16 个权重系数全部由作者手工设定。early 侧重 centering + upright，late 侧重 contact + both。两阶段划分（以 `training_progress` 为界）是线性的。
- **应有实现**: Agent 通过多轮实验观察不同权重配置下的训练效果，**自主搜索**权重空间，自主决定阶段划分方式（可能不是两阶段线性插值，而是更复杂的分段函数甚至 meta-controller）。权重曲线是 Agent 优化出来的，不是人工试出来的。

### 6. LunarLander 的 FDRE "进化" 未使用 LLM — `suite.py:68-73`

**文件**: `code/src/llm_reward_evolver/suite.py`
**方法**: `ExperimentSuite._run_fdre()`

```python
def _run_fdre(self) -> MethodResult:
    if _is_lunarlander(self.config.env_name):
        return self._run_seeded_reward_pool("fdre", LUNARLANDER_FDRE_CANDIDATES, ...)
    # 其他环境走这里
    return self._run_seeded_evolver(...)
```

- **当前实现**: LunarLander（主实验环境）被硬编码绕过 LLM 迭代进化，改为从 4 个作者手写的候选函数中训练一遍、选分最高的。这不是 evolution，这是 grid search over 4 hand-written candidates。
- **应有实现**: 所有环境统一走 Agent 自主进化管线——Agent 生成候选、训练评估、分析反馈、迭代改进。候选函数是 Agent 自己生成的，不是作者预先写好的。

---

### 7. LLM 角色 — 单轮代码生成器，非 Agent

**文件**: `code/src/llm_reward_evolver/llm.py`

```python
class LLMClient(Protocol):
    def complete(self, prompt: str) -> str: ...
```

整个 LLM 调用就是一个 `complete(prompt) → str`。

缺乏的能力：

| 能力 | 当前 | 应有 |
|------|------|------|
| Tool Calling | ❌ 无 | 应可调用 `train_agent`、`evaluate_model`、`read_training_curve`、`compare_experiments`、`analyze_trajectory` 等 |
| 多轮对话 | ❌ 无状态单次调用 | 应可在实验中持续对话，边看结果边调整策略 |
| Memory | ❌ feedback 字符串拼接，无持久记忆 | 应跨迭代积累经验：什么策略上次好使、什么环境该用什么结构 |
| 规划 | ❌ 外部 for 循环控制 | Agent 应自主决定：当前该探索新方向还是精调现有参数 |
| 读日志 | ❌ 不会 | Agent 应能读取训练日志、状态轨迹、评估曲线，从数据中形成判断 |

## 可复用模块

以下部分与硬编码解耦，可直接用于 Agent 系统：

- **`reward.py`** — 奖励函数安全沙箱：AST 语法校验、禁用危险调用（import/exec/eval/open）、运行时异常回退、reward clipping。通用且必要。
- **`wrappers.py`** — Gymnasium 奖励替换 wrapper（`CustomRewardWrapper`）。通用。
- **`trainer.py`** — PPO/DQN 训练 + 评估流水线。通用。

## Agent 系统应有架构

```
Agent 启动
  │
  ├─ Tool: env.reset() + 随机探索 → 收集状态轨迹
  ├─ Tool: inspect_observation() → 推断每个观测维度的物理语义
  │
  ├─ Think: "这是一个着陆任务。关键子目标：位置收敛、姿态稳定、速度衰减……"
  ├─ Tool: write_reward({子目标定义 + 初始系数})
  │
  ├─ Tool: train_agent(total_steps=50000)
  ├─ Tool: evaluate()  → score=50
  ├─ Tool: analyze_trajectory() → "episode短、触地前速度过大"
  │
  ├─ Think: "垂直速度惩罚不够，增加 slow 权重"
  ├─ Tool: refine_reward()
  │
  ├─ Tool: train_agent() → score=120
  ├─ Think: "进步但左腿不触地，加 contact 奖励"
  │
  └─ ... 循环直至收敛
```

**每一步决策是 Agent 自主做出的，领域知识是通过工具交互发现的，不依赖任何硬编码的环境规则或手工设计的子目标。**

---

## 结论

当前交付代码的价值在于证明了 HRDC（分层奖励分解 + 阶段动态权重）这个**结构设计思路**是有效的，以及提供了安全校验、训练管线等工程基础设施。但从 LLM Agent 自动化的视角看，本文主实验（LunarLander-v3）的所有关键决策——环境语义理解、子目标分解、权重系数设定、诊断反馈生成、候选函数构造——均由作者的人工领域知识完成，而非 LLM 自主发现。

要达成"LLM Agent 零人工干预自动进化奖励函数"的目标，需要在当前代码的工程管线基础上，将所有硬编码的 if-else 替换为 Agent 的 tool-use + multi-turn reasoning + persistent memory 架构。
